# Is there a future for Delivery Hub? — a grounded read (2026-07-06)

*Written after a full-day gauntlet: 4 fresh orgs cut, the loop driven with real records, and 4 independent code audits (feature-flags, runtime breakage, permset completeness, architecture). Everything below is backed by verified evidence, not vibes. Where it's a judgment call, it says so.*

## The verdict

**Yes — and the product is not the problem.** The honest one-liner: *Delivery Hub works; what's been failing is the install toolchain and a handful of permission/setup bugs that made a working product feel broken.* Today those got isolated, named, and mostly fixed.

David said it first and the evidence agrees: *"Delivery Hub in its current state would serve a customer. It's close. If it was you running the show, you could provide value."* The gap is **customer acquisition**, not capability.

## What "I don't know what's real" actually was

Three things were lying to you — none of them the product:

1. **The install toolchain.** `cci` is broken (auth bug); `sf` source-tracking fails on Windows while printing "Succeeded." → **Fixed:** one reliable install path, wrapped in `scripts/spin-up-demo-org.sh`. One command now cuts a fully-seeded, clickable org. This is the thing you lost when you lost dev help; it's back, and it's a script you own.

2. **Field-level security.** Metadata-deployed fields get zero visibility by default — so a fully-installed field reads as "missing" (SOQL: *No such column*; the demo seed: *Field does not exist*). A prior session read this as "the tooling drops fields / lies for months." It was a permission gap. → **Fixed:** 69 loop-field permissions backfilled; the exact seed that couldn't run now runs to completion.

3. **Two real first-run bugs.** The "Connect to Mothership" step NPE'd on a fresh org, and the admin health dashboard was dead under the permset. Both were the *actual* "wheels fall off when I use it." → **Fixed** this session.

Everything else the breakage audit checked — the daily intake → board → approve → invoice loop — came back **clean**: no invalid field refs, no broken navigation, no exceptions hiding failures. The loop is production-solid, with real tests behind every node.

## The one thing to verify before selling — ✅ checked (permission level)

The **SIZE** step (creating/sizing a work item) was the single highest-risk node — org config, not code. **Verified 2026-07-06:** the buyer permset `DeliveryHubApp` grants `WorkItem__c` **Create + Read**, and **every create-modal field including the size field `DeveloperDaysSizeNumber__c` is FLS-granted.** A buyer can submit and size a work item at creation. Risk node is **green**.

**One product decision (not a bug):** the buyer permset is **create + read only (`allowEdit: false`)** — a buyer cannot edit a work item after creating it (move cards, re-size). This reads as a deliberate *client-submits / vendor-delivers* split (the vendor operates via the admin app with full edit). If your intended model is a single operator running the whole board for a customer, that operator uses the **admin** app and is unaffected. Confirm the split matches how you'll sell it before flipping `allowEdit` on the buyer permset.

## The product strategy is already built

David's plan — *minimum features to get paid, flag off the rest, release as they cook* — is **already how Delivery Hub is architected.** Every feature is a datetime flag, OFF by default; the installer lights up only 2 internal ones. You don't have to build the flag layer. You have to **choose the cut:**

**The minimum paid loop:** request in → size → log hours → confirm done → cut invoice → record payment. Objects: WorkItem, WorkLog, DeliveryDocument, DeliveryTransaction. **Flags ON: essentially none** (a manually-cut invoice needs no flag). Then flip `HideTab*`/`HideHome*` ON to strip the surface to Board + Intake + Timeline + Documents — the same declutter already done for Jose. Everything else (AI, Watcher, Cart, approvals, forecasting, notifications, cross-org sync) stays dark until a paying customer asks.

Cleanup worth doing so the settings screen doesn't promise dead features: 2 flags (`EnableAIEstimation`, `EnableAutoGenerateDescriptions`) are surfaced in the UI but gate nothing in Apex — wire up or remove.

## Where nimba fits (the David conversation)

The one piece you genuinely can't do alone is what you named: **standing up orgs.** Today's script proves it's automatable; nimba (your Node port of cci) is exactly the layer that turns "run this bash script and pray" into "press a button." That's a concrete, honest reason for David to lean in — his tool solves your real bottleneck — rather than a favor.

## Honest open items (not blockers)

- 2 minor deleted-record `QueryException`s (`sendRequestToVendor`, `updateDocumentStatus`) — edge cases, one-line guards, un-fixed.
- Tier-1 FLS: 17 more fields (bounty/audit/sync objects, outside the paid loop) are invisible to everyone — pure bugs, safe to grant, not urgent.
- Cross-org sync (hub↔client) not driven end-to-end this session — it's the two-org MF↔vendor topology, optional for a single-org customer.
- The buyer-permset SIZE check above — do this before selling.

## Bottom line

You have not been building "more incomplete features." You built a working delivery platform whose *installer and permissions* were quietly broken, which made the whole thing feel fake. That's now diagnosed and largely fixed, with a repeatable install and a clear minimum-sellable cut. **The future of Delivery Hub is not a question of whether it works — it does — but of getting it in front of one paying customer.** That's a sales problem, which is a better problem to have than the one you thought you had this morning.
