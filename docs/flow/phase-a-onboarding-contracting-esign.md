# Phase A — New-client onboarding · contracting · document generation · e-signature
> Grounded read-only audit of the DH package (`delivery__`), 2026-06-17. Part of the end-to-end
> DH operating documentation + Cowork validation set. Honest as-built — flags what's broken.

## 1. AS-BUILT FLOW
**Onboarding entry:** `DeliveryClientOnboardingController.onboardClient(name, email, hourlyRate, phone, address)` (global @AuraEnabled) in one txn: validates (name+email req, rate>0, rejects dup `NetworkEntity__c.Name` exact/case-sensitive) → creates `NetworkEntity__c` (EntityTypePk='Client', StatusPk='Active', rate, contact) — **does NOT touch any cross-org connection fields** → generates a `Client_Agreement` doc → provisions `PortalAccess__c` (Admin role, 128-bit token) → returns {entityId, documentId, publicToken, portalAccessId, accessToken}.

**Doc generation:** `DeliveryDocGenerationService.generateDocument` → builds a **frozen JSON snapshot** (entity + issuer block [first Vendor/Both entity, else literal "Your Organization"] + work items + worklogs + requests) → injects 8 default agreement clauses incl. **Confidentiality (c4)** + **IP-assignment (c5)**, Governing Law = **Florida (hardcoded)**; Compensation clause varies Hourly/Retainer/Fixed_Bid → versioning/supersede → caps snapshot to field limit → **SHA-256** `DocumentHashTxt__c` over the snapshot → inserts `DeliveryDocument__c` (Draft, 256-bit public token, Net 30) → creates signature slots from `DocumentTemplateSlot__mdt` (Client_Agreement = Consultant[vendor] + Client[customer]) → publishes `DeliveryDocEvent__e`.

**Delivery:** PDF via `DeliveryDocumentPdf` VF page + `DeliveryDocumentPdfController` (`?id=` auth or `?token=` public; renders from frozen snapshot, live-rebuild fallback; flips Sent→Viewed on public GET). Email via `DeliveryDocEmailService.sendDocumentEmail` (renders PDF, CCs from settings, sends from verified OWEA, flips→Sent + due date; scheduled-send drained by `DeliveryHubScheduler` every 15 min).

**E-signature:** `DeliveryDocActionService.signActionInternal` shared by admin (`signActionAdmin`, **gated OFF** by `EnableAdminSigningDateTime__c`) and public-token (`signActionPublic` Aura + `DeliveryDocActionRestApi` `/sign/*`). Flow: validate consent+name → drawn sigs → ContentVersion PNG → `FOR UPDATE` lock (double-sign guard) → flip slot Completed, stamp consent/IP/UA, **null the one-time token** → audit `ActivityLog__c` → when 0 Pending slots remain AND doc ∈ {Sent,Viewed,Awaiting_Signatures}, advance doc→Approved + insert `DeliveryTransaction__c` (Approval). Status model: Draft→Ready→Sent→Viewed→Awaiting_Signatures→Approved (+Disputed, Superseded). Certificate-of-completion via `DeliveryDocCertificateService`.

**LWCs:** `deliveryClientOnboarding`, `deliveryDocumentSignPortal` (guest), `deliveryDocumentSignatureBlock`, `deliverySignaturePad`, `deliveryDocumentViewer`.

## 2. USER STEPS
1. Open `deliveryClientOnboarding` → New Client → name/email/rate (+phone/address) → Submit → toast + result panel. 2. View/Send agreement. 3. On the doc, confirm/assign the 2 signature slots. 4. (If admin-signing enabled) sign a slot: name + consent + optional draw. 5. Send (→ Sent) or copy a per-slot signer link. 6. Remote signer opens `?token=` page → review → name + consent + optional draw → Sign. When all slots done → Approved. (Portal "Approve" on a sign-required doc → Awaiting_Signatures, can't bypass signing; "Dispute" → Disputed.)

## 3. GENERALIZABLE vs MF-SPECIFIC
Generalizable: all Apex/LWCs/VF, 5 DocumentTemplate__mdt + 4 slot configs, crypto/audit, consent text; issuer identity is **data-driven** (first Vendor/Both entity). Florida governing law is a baked default. MF/subscriber config (data): the Vendor NetworkEntity (issuer branding), `DeliveryHubSettings__c` (DefaultBillingEntityIdTxt, DocumentCcEmailTxt, DocumentFromOrgWideEmailTxt, EnableAdminSigningDateTime), a verified OWEA/DKIM domain, a Site/Experience host for the public signer pages, per-client entity rows + rates.

## 4. GAPS / CAVEATS (honest)
1. **🔴 Hash-chain VALIDATOR can never reconcile.** `setHashOnInsert` (before-insert) hashes with `rec.Id` (null pre-insert) + `Datetime.now()`; `countBrokenLinks` validates with the real persisted `rec.Id` + `rec.CreatedDate` → recomputed hash never matches → validator reports every link broken. The trail *displays* (stored prior-hash read verbatim) and the doc SHA-256 is fine, but the advertised *tamper-detection validation* is non-functional. **Biggest soft spot in the e-sign claim.**
2. **🔴 No packaged Site / Experience Cloud / guest host.** Only VF pages + a guest permset ship. Public signing/portal does NOT work after a bare install — subscriber must stand up a Site + assign guest permset + surface the page. (Sign DML runs SYSTEM_MODE so the permset isn't the blocker — the missing host is.)
3. **🟠 NDA / non-compete / non-disclosure DO NOT EXIST as templates.** Only Client_Agreement, Contractor_Agreement, Invoice, Security_Audit, Status_Report. Confidentiality + IP exist only as inline clauses inside the two agreements. A standalone NDA/non-compete needs new metadata.
4. **🟠 Cross-org NetworkEntity registration is NOT part of onboarding.** Onboarding creates a Client entity but populates none of the connection fields (ConnectionStatus/ApiKey/HmacSecret/Endpoint/OrgId). The connection lifecycle is separate sync machinery — "establish the cross-org connection" is a disjoint flow.
5. Admin signing OFF by default + email gated on DKIM/OWEA → on a fresh install neither signing path nor email works without config.
6. IP captured only on REST sign path (Aura path leaves it null); consent timestamp == signature timestamp; `portalSessionEmail` hacked into the User-Agent (TODO for a real field); onboarding always uses a 1-yr term and never auto-dispatches the agreement.

## 5. PROS / CONS + ENHANCEMENTS (unscoped)
**Pros:** clean service/controller separation; immutable frozen-snapshot + hash design; real concurrency safety (FOR UPDATE + one-time token); versioning/supersede; field-cap trimming; metadata-driven templates/slots; shared signature LWC (touch + scaling).
**Cons:** the non-validating hash chain (the marquee compliance feature); no public host; brittle "first Vendor by CreatedDate" issuer selection (TODO for a branding setting); hardcoded Florida + English clauses despite a JurisdictionTxt field; UA-email hack; onboarding does too much in one opaque method; synchronous PDF email.
**Enhancements:** fix chain hashing to be deterministic (hash persisted fields / store the write-time stamp / move after-insert); ship/document an Experience site so signing works on install; add NDA/non-compete templates + clauses; branding block in settings; jurisdiction-driven governing law; dedicated PortalSessionEmail field + separate consent timestamp; optional auto-dispatch on onboarding; async PDF email + signer reminders.
