# Phase B — Business-process ingestion / definition (Delivery Hub, `delivery__`)
> Grounded read-only audit, 2026-06-17. The process-definition layer = a Custom-Metadata-driven workflow engine. A defined workflow IS the instruction set agents follow.

## 1. AS-BUILT FLOW
**Definition (all CMT):** `WorkflowType__mdt` (the process; 7 ship: Software_Delivery, Loan_Approval, Change_Management, Implementation_Project, Onboarding_Checklist, Operations, Support_Triage) → `WorkflowStage__mdt` (92 rows; `ApiValueTxt__c` = the value `StageNamePk__c` must equal; phase, colors, persona, flags Terminal/Blocked/Attention/VisibleByDefault as DateTime-bools, EtaWeight, and the transition graph `AllowedForward/BacktrackTransitionsTxt__c`) → `WorkflowPersonaView__mdt` (board columns per persona) → `WorkflowStageRequirement__mdt` (entry gates; only 1 ships: QA_Entry_Requirements) → `WorkflowEscalationRule__mdt` (SLA actions; 10 ship). Single read authority: `DeliveryWorkflowConfigService` (getWorkflowConfig/Types/Templates + terminal/attention/phase helper sets; cross-type union excludes 'Resolved').
**Movement:** a `WorkItem__c` binds to a process via `WorkflowTypeTxt__c` (plain Text, default 'Software_Delivery') and sits in `StageNamePk__c`. Stage changes are generic record updates from `deliveryHubBoard.js`; **transition legality + required fields are checked client-side only** (advisory). The before-update trigger `addError`s ONLY for dependency-blocking + Gantt-field guards — NOT transitions/requirements. Escalation IS server-side+scheduled (`DeliveryEscalationRuleEvaluator`/`ActionExecutor`/`Job` via `DeliveryHubScheduler`). Templates = a WorkItem with `TemplateMarkedDateTime__c` set (managed by `deliveryTemplateManager`; clone via `DeliveryRecurringItemService.cloneFromTemplate`).

## 2. USER STEPS
Author via `deliveryWorkflowBuilder` LWC (must be manually placed — package ships no FlexiPage/tab hosting it): create type → add stages (label/phase/color/flags/forward-transitions) → Save & Deploy (Metadata API enqueue → creates the type + stages + **Admin-only** persona views, async). Then **hand-author** rich config in Setup → Custom Metadata (multi-persona views, backtrack, ETA, requirements, escalation — builder can't). Then **Object Manager → Work Item → `StageNamePk__c` → add each new stage value** (builder skips this — without it items can't enter the stages). Set default via `DefaultDateTime__c`. Templates: mark a WorkItem `TemplateMarkedDateTime__c`.

## 3. GENERALIZABLE vs MF-SPECIFIC
Generalizable: the whole CMT quartet + config service are process-agnostic; persona board collapsing, escalation, attention/terminal/blocked, phase rollups, ETA weighting are config-driven; the 6 non-SD workflows are reference templates. SD/MF-specific leakage: `StageNamePk__c` inline picklist = the 37 SD values only (the operative coupling); `WorkflowTypeTxt__c` defaults SD; 'Deployed to Prod' ship-loop + QA requirement are SD-flavored; builder Phase dropdown is SDLC vocabulary; WorkItem schema carries dev/Gantt fields meaningless to non-dev.

## 4. GAPS / CAVEATS (honest)
1. **🔴 A non-dev process CANNOT run end-to-end — only display.** `StageNamePk__c` inline picklist = SD's 37 values; the CM/LA/IP/OP/ST/OC stage values are ABSENT (verified count=0). `deployWorkflow` creates WorkflowStage rows but never touches the picklist → a WorkItem can't be saved into a non-dev stage. The 7 non-SD workflows are demo/reference, not operational. **Architectural ceiling on "ingest any business process."**
2. **🟠 No referential integrity on process binding** — `WorkflowTypeTxt__c` is plain Text, no FK to `WorkflowType__mdt`; typos silently orphan items.
3. **🟠 Builder is a starter-generator (~30% of the model)** — emits only type + stages + forward transitions + Admin persona; hardcodes OwnerPersona=Admin, SortOrder=100; can't author backtrack/ETA/colors/multi-persona/requirements/escalation. Re-deploying an existing workflow LOSES hand-authored richness.
4. **🟠 Transition legality + stage requirements are advisory/client-side only** — any Apex/Flow/API/data-loader write bypasses the process entirely (verified: Apex update to an illegal stage succeeds).
5. Builder + template-manager LWCs are `isExposed=true` but **no shipped FlexiPage/tab places them** (recurring "subscriber must place surfaces" lesson).
6. Subscriber-org risk (flag): `deployWorkflow` creates managed-namespace CMT at runtime in the subscriber — known managed-package constraints; "works in dev/scratch, risky in subscriber" until verified live.
7. Phase vocabulary hardcoded SDLC; only 1 requirement + thin escalation set ship (latent capacity, not shipped behavior).

## 5. PROS / CONS + ENHANCEMENTS (unscoped)
Pros: clean definition/render separation, zero hardcoded process logic in the read path, genuinely multi-process/multi-persona, real scheduled escalation, expressive transition/flag semantics, templates reuse WorkItem schema. Cons: picklist coupling makes "ingest any process" aspirational; builder authors ~30%; no FK integrity; UI-only enforcement; every process edit is a metadata deploy; SDLC vocabulary leak.
Enhancements: make `StageNamePk__c` a GVS (or drive off unrestricted text) + have deployWorkflow extend the value set in the same deploy (closes #1); FK/validation on WorkflowTypeTxt (closes #2); promote transition+requirement enforcement into the before-update trigger, config-gated (closes #4); expand builder to author personas/backtrack/requirements/escalation + config-driven phases; ship a "Process Admin" FlexiPage hosting builder+templates (closes #5); an "ingest" path parsing a structured/NL process spec into the CMT quartet in one txn (the true ingestion engine the goal implies); per-type domain field sets.

## 7. AGENT OPERATING INSTRUCTIONS (instruction-engine)
The defined workflow = the agent's instruction set. **Watches:** stage deltas, time-in-stage vs escalation thresholds, attention/blocked stages, missing required fields. **Reads (safe):** getWorkflowConfig(T) = the instruction graph; terminal/attention/phase sets; requirements; escalation rules; templates. **Decision rules (deterministic):** move S→S' legal iff S'∈forward∪backtrack(S); enter S' only if required fields populated; overdue iff past threshold (honoring priority/no-response/cooldown); effort-terminal items consume no forecast. **Automatable today:** trigger escalation job (server-side), clone templates, read config, compute next-legal-move/what's-missing/overdue, side-effect stamping; the auto-diagnose pattern (anomaly→auto-drafted queue request, gated OFF) is the autonomous-drafting model. **MUST route to human:** authoring/changing the process (deployWorkflow / any CMT change — metadata deploy, packaging risk, changes everyone's rules); picklist value adds; any forced/illegal transition (platform doesn't enforce, so the agent must treat config as binding and surface violations, not force them); approval-gated transitions. **Errors:** INVALID picklist → route to admin to extend StageNamePk__c (don't retry); getWorkflowConfig exception → orphaned/typo'd type; builder deploy is async → poll status; empty-CMT defaults → "process not yet configured."
