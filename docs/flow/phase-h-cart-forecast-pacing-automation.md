## 1. AS-BUILT FLOW (grounded, cite code)

### H1 — Cart / Checkout / "the list"
The cart is **not a stored object** — it's a query predicate over `WorkItem__c` (`DeliveryCartService.queryCartItems()`):
```
cart = WorkItem__c WHERE ClientIntentionPk__c IN ('Will Do','Sizing Only')
       AND ActivatedDateTime__c=NULL AND ArchivedDateTime__c=NULL AND TemplateMarkedDateTime__c=NULL
```
`ClientIntentionPk__c` (inline picklist, 4 values): **Will Do** = committed → checkout candidate; **Sizing Only** = priced/exploring, not checked out; **Deferred / On Hold** = out of cart. Two LWCs: `deliveryCartBuilder` (the *add* surface — freeform-adds a WorkItem as Sizing/Will Do, running Σhours/Σ$) and `deliveryCartCheckout` (review + the **Checkout** button). Writes go through `DeliveryCartService` (FLS-guarded): `createCartItem` inserts a `WorkItem__c` (`Status='Active'`, `Stage='Backlog'`, `ActivatedDateTime__c` left NULL — it sets `DeliveryWorkItemTriggerHandler.triggerDisabled=true` in a try/finally to suppress the before-insert auto-activation that would otherwise eject the new line). `addToCart` promotes Sizing→Will Do; `removeFromCart` sets intention='Deferred' (not a delete); `reorder` writes `PriorityPk__c`. **`checkout()`** queries the Will-Do non-activated lines, **stamps `ActivatedDateTime__c=now()` on each** (the one load-bearing effect — lines cross into the live pipeline with no copy/convert), then branches on `CartExperienceProfile.checkoutMode`: **Invoice** (default) → conditionally generates a `Client_Agreement` doc; **Stripe** → activate only + "pending integration seam" message; **Approve** → activate only + "pending seam" message.

### H2 — Forecast / Pacing
There are **THREE independent forecast engines**, each with its own model, contract, and surface — this is the central confusion in this lane:
- **Engine A — Capacity slider** (`DeliveryHoursAnalyticsController.getForecastItems()`, non-cacheable, imperative; LWC `deliveryCapacityForecast`). The buyer "set the pace" 1–6 dev slider. The `buildSchedule` packer runs **fully client-side**: ramp `min(1, 0.45+mi*0.275)`, ceiling `devs×HOURS_PER_DEV(160)`, output floor `max(20, effCeiling−MAINT(90))`, **greenlit drains first**, build-out packs by start-date then size, run-point taper `max(0.10, 0.35−mi*decline)`, 18-month horizon → "Everything lands by {month}". `ForecastItemDTO` **drops assignee** (buyer-facing, no names).
- **Engine B — Portfolio pacing** (`getPortfolioPacing`, **cacheable**, backs the only `@wire`; LWC `deliveryPacingForecast`). Retrospective logged-vs-target SVG card: leaf-only counting (the 0.269 double-count fix), amortized target polyline, schedule-spread forecast bars. **Undated or overdue items → `unscheduledRemainingHours`, never a bar.**
- **Engine C — NG-rendered stacked pacing** (`getPacing`, non-cacheable; LWC `deliveryProFormaTimeline` feeds the vendored nimbus-gantt bundle). The `PacingData` JSON contract: per-bucket drill-down, stacked tier segments (Committed/Predicted/Ready/Recurring), `scopeLabel`, team pool. **This is the only consumer of `getTeamPool` and `getPacing`.**

Tier resolution is the link to approval: `tierForItem()` → `ClientPreApprovedHoursNumber__c > 0` ⇒ **greenlit** (green, drains first), regardless of lane; else lane-derived (`predicted`/`ready`/`recurring`, never greenlit).

### H3 — Automation orchestration
The orchestrator is **`DeliveryHubScheduler`** (`global without sharing implements Schedulable`). `scheduleAll()` registers **4 CronTriggers** (`Delivery Hub Sync - :00/:15/:30/:45`, every 15 min). Each `execute()` tick runs a flat ordered sequence of ~17 fault-isolated operations (each in its own try/catch — one failing job never aborts the tick, **and never surfaces an alert**): sync requeue/stage-heal/pending-resolve, escalations, **recurring-item cloning** (feeds the recurring forecast tier), poll, digests, invoice-gen, overdue reminders, **forecast alerts**, watcher digest, scheduled doc sends, reconciliation, **ETA auto-recalc** (writes the `EstimatedStart/EndDevDate__c` the forecast spreads against), activity-log cleanup, and **`runAutoDiagnoseIfEnabled()`** — the autonomy spine. The cart→approval→forecast chain: cart `submitForApproval` → auto-path (under `DiscretionaryThresholdHoursNumber__c`) sets `WorkItem.ClientPreApprovedHoursNumber__c=quoted` + `Stage='Ready for Development'`; human-path → `Status='Offer Sent'` → buyer `approve()` sets the cap. Setting the cap is what flips the forecast tier to greenlit. **Auto-diagnose** (`DeliveryAutoDiagnoseService.runSweep`) reads `ActivityLog__c` Field_Change rows (the SEE stream), counts anomalies per `DiagnoseRule__mdt` window, and when threshold crosses, drafts a `WorkItem__c` (`[auto-diagnose:<rule>]`) and **submits it into the same approval queue** — the literal "anomaly → human-approvable fix waiting, no human in between."

## 2. USER STEPS
**Cart (admin/MF default):** DeliveryHubAdmin app → AdminHome → Cart Builder: type desc + hours → "Add to size" (Sizing) or "Add & commit" (Will Do); per-line ✓ commit / priority / ✕ remove → Checkout Cart card (or Cart tab) → **Checkout**. **Forecast (buyer):** Delivery workspace → **Forecast** tab → drag 1–6 dev slider (recomputes instantly, no Apex hop) + Per-month/Cumulative toggle. **Pacing (Home / Velocity tab):** `deliveryPacingForecast` granularity + horizon picker. **In-gantt:** Timeline → Pacing subtab (NG stacked render). **Automation (one-time, per org):** run `delivery.DeliveryHubScheduler.scheduleAll();` once; seed `TeamPoolTxt__c` + `RecurringHoursPerMonthNumber__c`; flip the relevant `Enable*DateTime__c`; author `DiagnoseRule__mdt`; deploy the SEE Flow; set `EnableAutoDiagnoseDateTime__c`.

## 3. GENERALIZABLE vs MF-SPECIFIC
**Generalizable (ships in package):** cart-as-WorkItem-state (zero new objects); the entire pacing/forecast Apex (all derived from WorkItem/WorkLog/Settings); leaf-only counting; nameless `getTeamPool` fallback (`'Delivery Team|Delivery|160'` — never leaks the vendor roster); the whole scheduler tick; the cart→approval→greenlit chain; the generic `DiagnoseRule__mdt` engine; all `Enable*DateTime__c` gates. ## 4. HONEST GAPS — "features built but not used / wired to the wrong thing" (THE CORE)
This is the ledger of Glen's lived frustration, code-cited. Each line is a Cowork test and an intent-engine candidate.

**Cart:**
1. **`selfServe` is cosmetic — never enforced.** `EnableCartSelfServeDateTime__c`→`profile.selfServe` is resolved and shown in settings but **no LWC/Apex reads it to gate access, hide the builder, or scope the cart.** The "clients self-serve" toggle changes nothing.
2. **Stripe checkout = stub.** Returns "pending integration seam"; no Checkout Session / Payment Link callout exists. Setting mode=Stripe still just activates lines.
3. **Approve checkout = stub.** Returns "pending seam"; **no approval is submitted.** This is the most likely source of "checkout doesn't do anything."
4. **Invoice proposal silently degrades.** A doc only generates when committed lines share **exactly one** `ClientNetworkEntityLookup__c`, and `createCartItem` **never sets that field** — so any *freeform-built* cart structurally cannot produce a proposal. Checkout activates but produces no document; doc-gen exceptions are swallowed.
5. **The cart is org-wide (no "my cart").** `getCart` returns ALL non-activated cart WorkItems across the org; `selfServe` does not partition it.
6. **Builder placement not upgrade-durable.** `deliveryCartBuilder` lives only on `DeliveryHubAdminHome`; a subscriber-activated custom Home overrides it and survives upgrades (the validated workspace-tab lesson). The Cart *tab* (checkout) is durable; the builder is not.

**Forecast:**
7. **Three engines, three "done" dates.** Engine A ignores planned end dates (invents a timeline from `devs×160`); B/C spread against real `[start,end]`; the alert path uses velocity. The same WorkItem can show three different landing dates across surfaces.
8. **`TeamPoolTxt__c` does NOT feed the buyer capacity slider.** The slider ceiling is hardcoded `devs×160`; `getTeamPool` is consumed only by the in-gantt NG subtab. **The MF "seed TeamPoolTxt to de-sprawl the forecast" remediation has ZERO effect on the Forecast tab buyers land on.** This is the single most likely misroute in the lane.
9. **`unscheduledRemainingHours` is computed but never rendered** on `deliveryPacingForecast` — undated/overdue work silently vanishes from the pacing card (but the slider *packs* it → divergence).
10. **Synthesized bands are non-auditable.** `maintenance(90h)` + `orchestration` are constants with no field backing; a buyer can't reconcile the bar against the board (the card footnote admits "modeled reserves, not a board figure").
11. **Recurring tier silent unless seeded.** `RecurringHoursPerMonthNumber__c` defaults 0 → tier off, with no indication the lever exists.
12. **Forecast-alert sweep capped at 20 WorkItems** — a large portfolio under-scans silently (no batch chain yet).
13. *(hygiene)* `glenRate` persona name shipped in `deliveryCapacityForecast.js`.

**Automation:**
14. **The whole autonomy loop ships DARK, three prerequisites deep:** `EnableAutoDiagnoseDateTime__c` is null (OFF) on every org; **zero `DiagnoseRule__mdt` records ship**; and the events it reads only exist if an admin separately deployed the SEE Flow. All three are unsatisfied everywhere.
15. **Approval re-tiers but does NOT re-scope the forecast.** Scope keys on `ActivatedDateTime__c != NULL` + non-terminal stage — **not on approval.** Approving only flips the color/order; it does not add or remove an item. "I approved it and the forecast didn't change" is real (the item must also be board-active).
16. **Declined-after-approved stays green.** `decline` only parks an item when nothing was previously approved; a previously-greenlit item declined on an *increase* keeps its prior cap and stays Committed.
17. **Cap enforcement is FLAG-mode (advisory) everywhere.** `EnforceApprovalCapDateTime__c` is null on all orgs → over-cap work logs **save anyway** (status → Budget Hold + one bell). The greenlit cap the forecast shows is **not enforced at log time** — billing can exceed it silently.
18. **Silent-failure seams.** Every tick job swallows its exception into `System.debug` — no surfaced alert when a job (incl. the auto-diagnose sweep) fails. An agent IS the alerting layer.
19. **Once-per-day throttle.** Even fully ON, auto-diagnose / forecast-alerts / watcher evaluate once per GMT day; a same-day spike waits until tomorrow.
20. **Cron doesn't auto-schedule on install** — `scheduleAll()` must be run manually per org, or the entire tick never fires.

## 5. PROS / CONS + UNSCOPED ENHANCEMENTS
**Pros:** zero new schema across the whole lane; cart activation flows into forecast/gantt/sync with no convert step; honest seams (stubs say they're stubs); leaf-only fix kills the double-count; no names leak to buyers; slider reflows with no Apex round-trip; single fault-isolated orchestrator; conservative OFF-by-default install; non-cacheable forecast reads avoid stale-after-install.
**Cons:** two of three checkout modes non-functional; proposal unreachable from freeform carts; `selfServe` dead weight; org-wide single basket; three divergent forecast models; `TeamPoolTxt__c` doesn't reach the buyer surface; autonomy ships dark three layers deep; no surfaced failure alerting; cap advisory-only; approval doesn't re-scope.
**Unscoped enhancements:** enforce `selfServe` (gate builder + scope `getCart` to user); set `ClientNetworkEntityLookup__c` on `createCartItem` so proposals can fire; real WorkItem-basket approval object (MODE_APPROVE); Stripe Checkout Session callout; unify on ONE forecast engine (or feed the slider `getTeamPool` instead of `devs×160`); surface `unscheduledRemainingHours` as a "needs dates" callout; model MAINT/run-point as real fields + rename `glenRate`; move the alert sweep to Batch; **ship 2–3 starter DiagnoseRules (OFF)** so the loop is one flag away; **surface tick-job failures** as a WorkItem/bell; **post-install setup wizard** (schedule cron + list off-flags); **loop-health status tile** (last sweep, rules active, events flowing, flags on/off); approved-but-not-active reconciliation; make ENFORCE the default once verified.

## 7. AGENT OPERATING INSTRUCTIONS (the run-the-arena loop)
**PURPOSE:** keep the cart→approval→forecast→pacing chain healthy and the autonomy loop fed, so the owner only makes the deliberate approval decision.
**WATCHES:** the cart query (non-activated Will-Do/Sizing WIs); `WorkItem__c` active-scope (forecast input); Outbound `Failed`/`Staged`/`Queued` + Inbound `Pending`/`Failed`; `ActivityLog__c` Field_Change volume (auto-diagnose fuel); CronTrigger count.
