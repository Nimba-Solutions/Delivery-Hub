# Phase G — "The Machine": autonomy layer + the instruction engine for agents (Delivery Hub, `delivery__`)
> Grounded read-only audit + synthesis, 2026-06-18. North star (the-machine-0610): *a system that runs itself — agents learn the business, watch everything (SEE), turn inbound into staged ready-to-approve actions, catch + correct drift before anyone notices. Ladder: (1) owner is the integration layer; (2) owner becomes the approver — agents draft, human approves; (3) routine needs no approval. MCP exposes the platform as native tools.* This phase grounds that vision in code and tells the truth about the gap. **The spine is real and shipped but DARK; the machine sits at ~rung 1.5.**

## 1. AS-BUILT FLOW
Four-link chain: **SEE (capture) → ActivityLog (stream) → auto-diagnose (anomaly→draft) → approval queue (human gate)**, driven by the scheduler. All four exist in code; only the first ships configured.
**SEE:** two ingress paths writing `ActivityLog__c`. Trigger-side `DeliveryFieldTrackingService` (from DH object triggers, reads `TrackedField__mdt` allowlist, diffs old/new). Flow-side `DeliveryFieldChangeService.captureFieldChanges` = `global @InvocableMethod` "Capture Field Change" (globalized #918 so subscriber Flows feed ANY business object). Both persist a generic row: `ActionTypePk__c='Field_Change'`, `RecordIdTxt__c` (text — any object), `ComponentNameTxt__c`=field, `ContextDataTxt__c` JSON `{objectName, oldValue, newValue, delta…}`.
**Stream + tamper-evidence:** every row SHA-256 hash-chained on before-insert by `DeliveryAuditChainService.setHashOnInsert`; `validateChain`/`validateChainForRecord` walk it; `blockLegalHoldDeletion` addErrors under `LegalHoldDateTime__c`. **Caveat:** validator non-functional — insert hashes with null `rec.Id` + `Datetime.now()`, validate recomputes with persisted Id + `CreatedDate` → never matches → every link reports broken. Chain displays + is append-only; tamper-*detection* doesn't reconcile.
**Auto-diagnose ("Move 1"):** `DeliveryAutoDiagnoseService.runSweep()` — returns 0 if `EnableAutoDiagnoseDateTime__c` null OR `LastAutoDiagnoseScanDateTime__c`=today (once/day guard); queries active `DiagnoseRule__mdt` (those with `ActiveDateTime__c`); per usable rule skips if `alreadyDraftedWithinCooldown` (counts marker-titled WIs), else `countAnomalies` scans `ActivityLog__c` (Field_Change + field + window, cap 5000); if `hits >= CountThresholdNumber__c` → `draftRequest` inserts a marker-titled `WorkItem__c` + `DeliveryWorkApprovalService.submitForApproval` → **Offer-Sent in the existing queue** (or auto-accepted under the discretionary threshold). **Only drafts — never approves, never executes the fix. The queue is the hard stop.**
**Scheduler:** `DeliveryHubScheduler` (4 cron, idempotent `scheduleAll` from install handler). Each `execute()` runs ~18 sub-jobs incl. line-37 `runAutoDiagnoseIfEnabled()` (try/caught). *(.claude/worktrees copies are stale and omit this; the main force-app copy is authoritative.)* Most jobs hour/flag-gated.
**Watcher stream:** `DeliveryWatcherService.runAndReturnSummary` (daily, gated). 3 real signals (SLABreach, StuckStage, ARAging); 4 inert stubs (FailedSyncTrend, PaymentOps, UnhappySignal, UpcomingSignoff) return empty. 14-day stabilization gate + heartbeat are the only thing resembling confidence/self-gating.

## 2. USER / ADMIN STEPS (turn the machine on; gates = `DeliveryHubSettings__c` DateTime fields, null=OFF)
**A. Eyes:** stamp BOTH `EnableFieldTrackingDateTime__c` + `EnableActivityLoggingDateTime__c` (DH's own objects auto-tracked via 11 shipped `TrackedField__mdt` rows). **B. Point SEE at a non-DH object (no Apex; `docs/FIELD_TRACKING.md`):** add a `TrackedField__mdt` row + stamp `EnabledDateTime__c`; build a Record-Triggered Flow → Apex Action "Capture Field Change" → map Record ID/Object/Field/Old/New; confirm step A on. **C. Auto-diagnose (triple gate, ALL required):** stamp `EnableAutoDiagnoseDateTime__c`; author ≥1 `DiagnoseRule__mdt` (object+field+threshold+window+quoted hours+`ActiveDateTime__c`); ensure SEE is feeding events for the watched object. **D. Watcher digest (optional):** stamp master + per-signal flags + recipient IDs/Slack. **E. Scheduler bootstrap:** normally automatic; else `delivery.DeliveryHubScheduler.scheduleAll();`.

## 3. GENERALIZABLE vs MF-SPECIFIC
Generalizable (engine, not tenant): the whole capture engine + global invocable, `ActivityLog__c` + hash chain, `DeliveryAuditChainService`, `DeliveryAutoDiagnoseService`, `DiagnoseRule__mdt` (names no tenant — points at any object/field via config), `TrackedField__mdt`, scheduler, Watcher orchestrator + 3 real signals. The auto-diagnose comment is explicit: *"GENERIC by design — the package ships the engine, not the rule."* Config (per-org): the 11 shipped TrackedField rows, gate flags, DiagnoseRule records, Watcher recipients. MF-specific / not in package: **ZERO DiagnoseRule records ship**; the SEE Flow on any MF business object is built in MF's org; quote amounts, thresholds, which process to watch, Slack config = MF judgment.

## 4. GAPS / CAVEATS (how much of "the machine" is real)
**Real + shipped (spine exists):** ✅ SEE capture (built, globalized, ships configured for DH's 11 fields — most-real link); ✅ auto-diagnose engine (built, scheduler-wired, tested — anomaly→draft→queue is end-to-end in code); ✅ approval queue as human gate (`DeliveryWorkApprovalService` global, decision-complete; auto-diagnose reuses it 1:1); ✅ scheduler loops, escalation, recurring, reconciliation, Watcher's 3 real signals.
**Gated OFF (real but dark):** ⚠️ SEE writes nothing until both flags stamped; ⚠️ auto-diagnose behind a **triple gate** (master flag + active rule + a SEE stream feeding the object) — **with zero shipped rules it no-ops every tick out of the box. This triple gate IS the "what blocks full autonomy" answer.**
**Unbuilt / aspirational (vision outruns code):** 🔴 **MCP server = design doc only** (`docs/plans/dh-mcp-server-design-2026-06-12.md`); Apex half ready (global approval service), missing half = thin REST routes (gap G1: no decision routes) + TS server + per-user OAuth (G4). 🔴 **Confidence ladder / self-gating unbuilt** — no confidence score graduates draft→auto; only rate-limiters (Watcher 14-day gate, auto-diagnose cooldown). 🔴 **Autonomy ladder has no rung metadata** — encoded only as binary `Enable*DateTime__c` gates; `EnableAutoDiagnoseDateTime__c` is the single "approver→autonomous" lever and even fully on only *drafts into a human queue* — never reaches rung 3. 🔴 4 of 7 Watcher signals inert. 🔴 hash-chain validator doesn't reconcile. 🟠 anomaly model shallow (literal-value threshold count, no trend/correlation).
**Two ladders, don't conflate:** `FeatureDefinition__mdt.MaturityPk__c` (Alpha/Beta/GA — feature lifecycle, UI pill) ≠ autonomy. `deliveryScore`/`DeliveryScoreService` (0–100 pipeline-health, 6 factors) is not a maturity/autonomy meter and feeds no gating today (could be a future confidence substrate).
**Net:** spine is real + shipped but **dark** — eyes configured, brain wired, queue ready, but the one rule that makes MF's loop fire isn't authored, the confidence ladder doesn't exist, and MCP is a design doc. **~Rung 1.5:** can capture + draft, but a human must both *configure it on* and *approve every output*.

## 5. PROS / CONS + ENHANCEMENTS (unscoped)
Pros: clean separation (capture / stream / diagnose / queue independently testable + config-driven); genuinely generic (1 CMT row + 1 flag adds a watched process); safe-by-default (every link OFF until stamped; auto-diagnose self-rate-limits + dedups); one approval pipeline for human + machine; append-only attributed audit substrate. Cons: spine ships dark (config still = author rules + flip flags + build Flows); no confidence/graduation model (rung 3 unreachable by design); shallow anomaly detection; MCP unbuilt; hash validator broken; half the Watcher signals stubs; two unrelated "ladders" invite confusion.
Enhancements: add an autonomy-rung field (`AutonomyLevelPk__c`: Manual/Approver/Auto) so the ladder is data; build a confidence ledger (per-rule draft→approve→reject outcomes in `ActivityLog__c` → accuracy score → gate auto-promotion on N approvals); fix hash chain to hash deterministic persisted fields; ship the MCP server + G1 routes + per-user OAuth; promote the 4 stub signals + feed Watcher findings into auto-diagnose; richer anomaly model (trend/delta/baseline); an inbound→draft path (parse email/Slack/webhook into a staged WI the way auto-diagnose drafts).

## 7. THE INSTRUCTION ENGINE — SYNTHESIS (the most important section)
### 7.1 What a real instruction engine for agents needs
An "instruction engine" is the canonical, machine-readable operating manual an agent reads **per process and per phase** to run the business. DH proves the *shape* in Phase B: *"A defined workflow IS the instruction set agents follow."* A complete operating instruction for any phase has **eight canonical parts**: (1) PURPOSE/outcome — what "done" means; (2) WATCHES (the eyes) — signals monitored; (3) READS (safe inputs) — config + state read with no side effect; (4) DECISION RULES (deterministic) — legality/threshold logic mapping state → candidate action; (5) AUTOMATABLE-NOW actions — unsupervised side effects (read config, compute next move, stamp, clone, escalate, **draft**); (6) MUST-ROUTE-TO-HUMAN — the hard stops (approve a quote, change the process, force an illegal transition, add a picklist value); (7) AUTONOMY RUNG — Manual/Approver/Autonomous for this action class; (8) ERROR/ESCALATION protocol. Phases A–F already authored parts 1–6 and 8 (their §7). **What's missing across the board is part 7 — the autonomy rung — because it has no home in the data model. That is the single biggest structural gap between "documented instructions" and "an instruction engine."**

### 7.2 How per-phase agent instructions plug into the maturity ladder
The vision's ladder (manual→approver→autonomous) is an *autonomy* axis, distinct from `MaturityPk__c` (feature lifecycle) and `deliveryScore` (pipeline health). Today the rung is encoded only as binary `Enable*DateTime__c` gates.

| Phase | Manual (rung 1) | Approver (rung 2 — agent drafts, human approves) | Autonomous (rung 3 — no approval) |
|---|---|---|---|
| Onboarding (A) | `onboardClient` by hand | Agent drafts NetworkEntity + agreement, human signs | *Unbuilt* |
| Process-def (B) | Admin authors via builder | Agent proposes stages/rules; human deploys CMT | **Never** (CMT = packaging risk) |
| Intake (D) | Human creates WI | **`DeliveryAutoDiagnoseService` drafts a WI into the queue** (only shipped rung-2 mechanism) | *Unbuilt* (no confidence promotion) |
| Quote/approve/deliver (E) | Human submits + approves | `submitForApproval`→Offer-Sent→human decides; **discretionary-cap auto-approve** is the closest to rung 3 | Partial rung-3 *only* under the cap |
| Billing (F) | Human runs invoice gen | Scheduler drafts invoices behind a flag | Flag-on = autonomous for a *fixed* recurring job (not learned) |

Pattern: **rung 2 is the design center of the whole package** — the approval queue is the universal "agent drafts, human approves" substrate, and auto-diagnose is the one mechanism that *produces* a rung-2 draft from a watched signal. **Rung 3 is reachable today only as a static flag or a fixed discretionary cap — never as a *learned* graduation.** No metadata says "this rule earned autonomy," so an agent cannot self-promote.

### 7.3 The concrete gap between today's pieces and an agent that runs MF's business
| Engine component | Concrete DH primitive (shipped) | Gap to "agent runs MF" |
|---|---|---|
| Eyes | SEE stream — `ActivityLog__c`, fed by `DeliveryFieldChangeService` + global invocable | Flip on + build a Flow per MF object; only DH's own objects pre-wired |
| Per-process instruction | CMT: `WorkflowType/Stage/Escalation__mdt`, `TrackedField__mdt`, `DiagnoseRule__mdt` | Non-dev process can't *run* (Phase B: `StageNamePk__c` coupling); **zero DiagnoseRule rows ship** |
| On/off | `Enable*DateTime__c` gates | Binary only — no rung field, no graduated autonomy |
| The draft | `DeliveryAutoDiagnoseService`→`submitForApproval`→Offer-Sent | Shallow anomaly model; no inbound→draft path |
| Human-as-approver checkpoint | Approval queue + discretionary auto-approve | Decision verbs not exposed via REST/MCP (G1) — agent can't *drive* the queue as a tool |
| Tamper-evidence | `DeliveryAuditChainService` SHA-256 chain | Validator doesn't reconcile (Phase A) |
| Attribution guardrail | `assertCallerMayDecide` (caller-is-approver) | Needs per-user OAuth for MCP (G4); an integration user attributes every decision to a bot |
| Agent-as-operator surface | — | **MCP server is a design doc only** |
| Self-gating / confidence | Watcher 14-day gate; auto-diagnose cooldown | Rate-limiters, not a confidence score; no draft→auto promotion |

**Honest summary:** MF could run a single, narrow, human-approved self-heal loop **today** (flip the SEE flags, build one Flow, author one DiagnoseRule, stamp the auto-diagnose flag) — that is rung 2 for one process. To get an *agent* "running the business," three things must be built that don't exist: (a) the **MCP tool surface** (agent reads the board + drives the queue without the `sf` CLI); (b) the **autonomy-rung + confidence ledger** (actions graduate approver→autonomous on evidence); (c) an **inbound→draft path** (email/messages/requests become staged actions, not just field-change anomalies).

### 7.4 The human-as-approver guardrail model (consolidated)
Rung 2 — "owner becomes the approver" — is the part DH guards best, one coherent model from four primitives: (1) **the queue is the universal choke point** — every machine-originated action flows through `submitForApproval` → Offer-Sent (visible, reversible, awaiting decision); the agent drafts, does not decide. (2) **Caller-is-approver attribution** — `approve`/`decline` enforce the running user via `UserInfo.getUserId()`; MCP design requires *per-user OAuth* so the agent inherits the human's authority, never mints its own. (3) **Discretionary auto-approve is the only sanctioned rung-3 escape hatch** — bounded by an explicit org cap, not agent judgment (human pre-authorized the *class*). (4) **Tamper-evident attributed audit** underneath (`ActivityLog__c` stamps the real user; caveat: validator defect). The guardrail design is sound and ready; what's missing is not the *gate* but the *graduation policy* (when may an action skip the gate?) and the *tool surface* (how does an agent reach the gate?). Build those two and DH's spine carries the vision to rung 2 in production and gives rung 3 a credible, evidence-gated path.
