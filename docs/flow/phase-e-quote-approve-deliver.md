# Phase E — Quote → approve → deliver → ship (Delivery Hub, `delivery__`)
> Grounded read-only audit, 2026-06-18. The decision lives ON `WorkRequest__c` (no child approval object). The estimate→offer half is agent-ready (and lacks a packaged UI anyway); the approve/decline half is deliberately human; delivery moves are CMT-gated; ship + cap-flagging are automatic with documented side effects.

## 1. AS-BUILT FLOW
Lifecycle `StatusPk__c`: Draft → Offer Sent → Accepted (or Inactive on decline; Budget Hold on over-cap). One request decided once; increases are *new* requests.
**A. Propose** `DeliveryWorkApprovalService.proposeEstimate(wi, hours, reasoning)`: stamps WI `EstimatedHoursNumber__c` + newest Draft's `QuotedHoursNumber__c` (mints Draft if none), writes reasoning as `WorkItemComment__c` `'Estimate proposal: '`, audits `Propose_Estimate`. Stays Draft.
**B. Submit** `submitForApproval(ids[, quotedByItem])`: one request per item; either **discretionary auto-approve** (quoted ≤ `DiscretionaryThresholdHoursNumber__c` AND month's auto-sum < `DiscretionaryMonthlyCapHoursNumber__c` → status **Accepted** + `AutoApprovedDateTime__c`, WI `ClientPreApprovedHoursNumber__c`=quoted, stage Ready for Development, audit `Auto_Approve`) or → **Offer Sent** + `ApproverUserLookup__c`, stage Ready for Final Approval, ping approver.
**C. Decide (queue LWC):** `approve(reqId, approvedHours, note)` → Accepted, WI `ClientPreApprovedHoursNumber__c` (canonical client cap), stage Ready for Development; increase raises cap by delta (never overwrites); null hours = approve-at-quote, non-null = counter. `approveMany(ids, note)` loops approve, collecting failures into `BulkApproveResultDTO`. `decline(reqId, reason)` → Inactive + reason; parks WI `On Hold` only if nothing previously approved. `requestIncrease(wi, extra, reason)` → NEW Offer Sent with `RequestedBudgetIncreaseNumber__c`; never auto-approves.
**D. Deliver** (`deliveryHubBoard`): stages from CMT (`DeliveryWorkflowConfigService`). Moves via move-modal (`updateRecord` on `StageNamePk__c` after `getRequiredFieldsForStage`) or drag-drop (CMT-validates → `reorderWorkItem` Apex). Board live-refreshes off `DeliveryWorkItemChange__e` (stage_change). In-flight band = `DeliveryApprovalSummaryController.IN_FLIGHT_STAGES`.
**E. Ship** (into `Deployed to Prod`, `DeliveryWorkItemTriggerHandler` 605–676): before-update `assignDeployVerifier` sets `DeveloperLookup__c`=approver (skips inbound-sync); after-update `notifyDeployedToProd` → one bell "N deployed — please verify". **"Deployed — Awaiting Verification" report** (tabular, `StageNamePk__c='Deployed to Prod'`, by StageEntered) = 4th summary-card tile.
**F. Cap at WorkLog** (`DeliveryWorkCapEnforcementService`, wired in `DeliveryWorkLogTriggerHandler`): sums logged vs `ClientPreApprovedHoursNumber__c`, two postures gated by `EnforceApprovalCapDateTime__c` (see §4).
**G. Forecast** `getPortfolioPacing` (history + forecast buckets, leaf-only, terminal-excluded, schedule-based forward spread) → `getPacing` (nimbus-gantt `PacingData` stacked tiers Committed/Predicted/Ready/Recurring); `getForecastItems` (flat leaf list, no names) → `deliveryCapacityForecast` slider client-side repacks against dev-count ceiling ("lands by").

## 2. USER STEPS
**Buyer:** Delivery tab → Approvals → summary card 3 tiles (approved-this-month / pending / in-progress) + pitch strip → queue (epic-grouped, oldest-first): Approve / Approve-with-change / Decline; bulk Approve selected/Approve all (N); "Why this estimate" expander. Forecast tab: pacing chart + capacity slider (devs 1–6 → "lands by", per-month/cumulative). **Delivery team:** estimate today has **no packaged button** (cart + Apex/agent/Flow calling the global methods); move work on the board; ship → Deployed to Prod (auto-verifier + ping); log hours (over-cap → flag flips Budget Hold + bell, or enforce blocks → requestIncrease → buyer approves → re-log).

## 3. GENERALIZABLE vs MF-SPECIFIC
Generalizable: entire approval service + DTOs (global, namespace-safe), cap enforcement (both modes), queue/summary/forecast LWCs, the 4 reports, CMT board stages/transitions, all settings; forecast uses ZERO new objects/fields. Backup-approver, discretionary auto-approve, recurring tier all off by default. MF-specific (data): approver user IDs (Jose `005UN…`), `RecurringHoursPerMonthNumber__c≈69`, `TeamPoolTxt__c`, and the ramp/maintenance/taper constants in `deliveryCapacityForecast.js` (synthesized bands ported from an MF prototype — apply to any org but calibrated to MF); the Software_Delivery pipeline shape.

## 4. GAPS / CAVEATS (honest)
1. **🔴 Cap defaults to FLAG (advisory), not block** — `EnforceApprovalCapDateTime__c=null` ⇒ over-cap logs save anyway (request → Budget Hold + bell only). Real blocking needs the DateTime set. Out of the box the "client-approved cap" is an *alert*, not a *limit*.
2. **🟠 Auto-approve status is `Accepted`, not `'Auto'`** — the `DiscretionaryThresholdHoursNumber__c` field description is stale; distinguish by the `AutoApprovedDateTime__c` stamp.
3. **🟠 Shipping clobbers the developer** — `assignDeployVerifier` overwrites `DeveloperLookup__c` with the approver on entry to Deployed to Prod; the ship report relabels it "verifier." Who built it is lost on that record.
4. **🟠 Approval scoping permissive** — `getPendingForApprover` returns requests assigned to the user OR unassigned (NULL approver); with no approver configured every authorized user sees/decides the whole queue. Not multi-buyer-safe in one org.
5. **🟠 No packaged UI to create estimates/offers/increases** — no LWC imports `submitForApproval`/`proposeEstimate`/`requestIncrease`; the only in-package caller is `DeliveryAutoDiagnoseService` (off by default). Estimating is API/agent/Flow-driven.
6. **🟠 `approveMany` does per-item DML in a loop** (bounded by the 200-row queue cap).
7. **🟠 Cap exempts inbound sync** (relayed logs never capped — correct, but caps only bind at the origin org).
8. **🟠 Forecast caveats:** schedule-based not velocity-based (`projected = logged + Σ max(0, est−logged)`); run-rate computed but doesn't drive the bars; **date-less / overdue-planned items go to `unscheduledRemainingHours` and produce NO bar** (stale dates → deflated curve while headline still counts the work); leaf-only assumes parent estimates are rollups; ramp/maintenance/taper are synthesized constants.
9. **🟠 Auto-approve monthly cap is best-effort within a txn** (concurrent submits could overshoot).

## 5. PROS / CONS + ENHANCEMENTS (unscoped)
Pros: decision-on-record (simple + auditable, SHA-256 hash-chain via ActivityLog); global API → agent/Flow-drivable; zero-new-field forecast; cap has a safe default + hard mode; queue epic-grouped + bulk; namespace-safe reports. Cons: advisory-by-default cap surprises; no estimate-creation UI; permissive queue scoping; developer-field clobber; stale field metadata; forecast silently hides date-less work; per-item approve DML.
Enhancements: one-click "Submit"/"Request increase" quick actions; per-org cap default + posture banner; preserve original developer on ship (separate `VerifierLookup__c`); surface `unscheduledRemainingHours` in the forecast card; per-buyer queue partitioning (NetworkEntity-scoped); fix the threshold field description; bulkify `approveMany`; make ramp/maintenance bands real fields.

## 6. COWORK TEST SCENARIOS (exhaustive — 69; Pre→Action→Expected→Verify). Sandbox first.
**Estimate/propose (1–6):** propose happy (Draft minted, comment, audit); reuses newest Draft; zero/negative rejected; blank reasoning rejected; null wi rejected; proposal note surfaces in queue.
**Offer/submit (7–15):** → Offer Sent (no discretionary); auto-approve under threshold (status Accepted NOT 'Auto', no approver ping); at exactly threshold (inclusive); just over (Offer Sent); monthly cap exhausted → Offer Sent; running cap within one batch (document actual); explicit map overrides estimate; empty set rejected; unknown id rejected.
**Approve (16–24):** at quote; with change (counter); increase raises cap (90 not overwrite); increase null hours → quoted; wrong user rejected; backup allowed; unassigned approvable by any (stamps decider); already-decided rejected; null id rejected.
**Bulk (25–30):** all succeed; partial (failures parsed); all fail; approve-all-in-epic (section only); stale selection drops dead id; select-all then approve.
**Decline (31–34):** fresh item → On Hold + ping; decline-an-increase (WI untouched); blank reason; wrong user.
**Increase (35–38):** happy NEW Offer Sent; never auto-approves; zero rejected; badge in queue.
**Stage moves (39–45):** advance valid (LDS); advance requiring fields (modal); backtrack (only if allowed); drag invalid blocked; drag valid + reorder (Apex); live refresh (stage_change only); My Work filter.
**Deploy/ship (46–51):** move into Deployed to Prod (developer overwritten to approver, bell); multiple in one txn (single bell); already-deployed re-save (no re-fire); inbound-sync deploy (skip assignment); no approver (silent no-op); ship report contents.
**Cap (52–59):** flag-mode over-cap (saves + Budget Hold + bell); enforce-mode over-cap (addError, not committed); enforce partial batch (3h blocked, 1h passes); uncapped item (never evaluated); update hours over cap (enforce blocks); update unrelated field (skipped); inbound-sync over-cap exempt; unblock loop (increase→approve→re-log).
**Forecast/pacing (60–69):** pacing renders; approve raises Committed tier (immediate, non-cacheable); slider reflow ("lands by" moves, greenlit drains first); cumulative toggle; date-less item invisible (no bar but in headline); terminal item excluded; recurring tier (69 → segment, 0 → absent); over-budget flag; summary tile == queue count reconciliation; empty portfolio empty-state.

## 7. AGENT OPERATING INSTRUCTIONS (instruction-engine)
The package is built so an instruction-engine drives the whole rung with a human at the **approval** gate. **Drafts an estimate:** `proposeEstimate(wi, hours, reasoning)` — real justification, never fabricated numbers (CLAUDE.md data-integrity). **Routes to human:** `submitForApproval({wi})` → Offer Sent + approver ping (don't assume auto-approve unless the threshold is set, and treat non-trivial estimates as human-gated; precedent = `DeliveryAutoDiagnoseService`, off by default). **Drives delivery:** move stages via record update on `StageNamePk__c` ONLY when the transition is in `forwardTransitions` and required fields are satisfiable — else route to human. **Pings on ship:** set stage Deployed to Prod (fully automatic verifier + bell + report) — aware it **overwrites `DeveloperLookup__c`** (capture the real dev first). **Increase loop:** on Budget Hold (flag) or addError block (enforce), draft `requestIncrease` and route to the queue. **Errors:** `AuraHandledException` messages are generic in managed package — branch on the *call*, never message text; `approveMany` returns failures (parse `BulkApproveResultDTO.failures`); notifications are fire-and-forget. **Automatable today:** propose (sourced reasoning), submit (route to human), request increase, read pacing/forecast to decide what to estimate, drive ship pings. **Needs human:** approve/approveMany/decline (buyer's money commitment); flipping `EnforceApprovalCapDateTime__c`; arbitrary backward/forced stage moves; overwriting a real developer without preserving it; MF-tuned data (approver IDs, recurring hours, team pool).
